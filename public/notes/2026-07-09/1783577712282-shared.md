## AI在前端工作中实践分享

整体工作流程：需求拆解 -> 页面&组件编码 -> 接口与逻辑开发 -> 排错&性能优化 -> 测试验证 -> 交付

## 各环节AI具体用法
### 1. 需求拆解
把产品原型/文案丢AI，输出：页面结构、组件拆分、接口字段、状态逻辑

### 2. 页面&组件编码
1. 描述样式、布局、交互，AI生成**Vue/React/TSX**基础代码
2. 批量生成列表、弹窗、表单、表格通用模板
3. 统一规范：命名、样式Less、校验规则

### 3. 接口与逻辑开发
- AI生成请求封装、入参出参处理
- 复杂计算、防抖节流、日期格式化直接生成工具函数
- 状态管理逻辑梳理+代码实现

### 4. 排错&性能优化
- 粘贴报错日志，AI定位bug、给出修复代码
- 检测冗余代码、闭包、渲染卡顿，给出优化方案
- 兼容性、移动端适配问题快速修复

### 5. 测试&校验
- 生成测试用例并结合Playwright完成测试验证，输出测试报告


## 具体样例

完成一个需求后，需要进行测试；这个用的Playwright MCP来做
Playwrigh：浏览器自动化测试框架。
关键特性：
1. 自动等待：操作前自动等元素可见、可交互、稳定，降低用例失败率
2. 强隔离上下文：每个测试独立浏览器上下文，可复用登录状态
3. 强大定位器：支持css/xpath/文本，自动重试，动态元素更稳
4. 网路拦与模拟：拦截、修改请求、模拟接口失败、mock数据
5. 调试利器：内置录制生成代码、Inspector可视化调试、trace快照、自动截图
6. 原生支持复杂场景：弹窗、多页面、iframe

Playwright MCP做了哪些事情？
LLM <--> MCP协议 <--> Playwright <--> 真实浏览器

启动并管理浏览器
把页面变成AI能看懂的“结构化快照”，不取像素截图，而是读浏览器无障碍树，输出精简文本快照；
把LLM指令翻译成浏览器操作（内部支持多种MCP，LLM可调用）
双向反馈循环：LLM发指令 -- MCP 执行 -- 返回新快照/结果  -- LLM继续决策，自主完成多步骤任务

Playwright MCP 本质上是**把 Playwright 包装成一个 MCP 服务器**，让大模型（LLM）能用**自然语言+结构化快照**直接操控浏览器，**不用写代码、不用视觉模型**。下面按“它做了什么 → 核心创新 → 提供的能力 → 解决的问题 → 典型用法”来讲清楚。

---

### 一、它到底做了哪些事（一句话概括）
Playwright MCP = **LLM ↔ MCP协议 ↔ Playwright ↔ 真实浏览器** 的桥梁。具体承担：

1. **启动并管理浏览器**  
   自动装 Chromium/Firefox/WebKit，可无头/有头，持久化上下文（登录态、Cookie）。

2. **把页面变成AI能看懂的“结构化快照”（最核心）**  
   不取像素截图，而是读浏览器**无障碍树（Accessibility Tree）**，输出精简文本快照：
   - 只保留：**角色（button/textbox）、文本、状态（checked/disabled）、唯一ref**
   - 示例：
     ```
     heading "todos" [level=1]
     textbox "What needs to be done?" [ref=e5]
     checkbox "Toggle Todo" [ref=e10]
     ```
   - 好处：**200–400 token/页**（截图要几MB），**100%精准定位**，不受前端样式改动影响。

3. **把LLM指令翻译成浏览器操作**  
   提供 **40+ 标准MCP工具**，LLM通过ref调用：
   - 导航：`browser_navigate`、`back/forward`
   - 交互：`click(ref)`、`fill(ref, text)`、`select`、`hover`
   - 取证：`screenshot`、`console logs`、`network intercept`
   - 高级：`evaluate JS`、`upload file`、`mock API`

4. **双向反馈循环**  
   LLM发指令 → MCP执行 → 返回新快照/结果 → LLM继续决策，**自主完成多步骤任务**（如登录→下单→截图）。

5. **统一协议，跨客户端可用**  
   支持 Claude Desktop、VS Code、Cursor、Windsurf 等所有 MCP 客户端，**一套能力到处用**。

---

### 二、核心创新：告别“看图说话”，改用“语义交互”
传统AI浏览器控制（截图+视觉模型）：
- 慢、耗token、识别不准、动态元素易失败

Playwright MCP：
- ✅ **基于语义（无障碍树）**，不是像素
- ✅ **无需视觉模型**，纯文本交互
- ✅ **确定性定位**，ref永久绑定元素
- ✅ **自动等待**（继承Playwright），元素就绪再操作

---

### 三、对外提供的能力（LLM视角）
- **浏览网页**：打开URL、前进后退、刷新
- **操作元素**：点击、输入、下拉、复选、悬停、拖拽
- **表单处理**：填写、提交、文件上传
- **数据提取**：读文本、表格、JSON-LD
- **调试取证**：截图、录屏、控制台日志、网络HAR
- **JS执行**：运行自定义脚本、获取/修改DOM
- **网络控制**：拦截请求、Mock响应、限速
- **多标签/iframe**：切换、操作内嵌页面

---

### 四、它解决了什么问题
1. **降低AI浏览器自动化门槛**：不用写Python/JS脚本，自然语言即可
2. **大幅节省token**：结构化快照比截图少**90%+**token
3. **更稳定**：自动等待+语义定位，减少“元素找不到”
4. **可解释**：每步操作基于清晰快照，便于LLM推理
5. **开箱即用**：一条命令安装，自动管理浏览器

---

### 五、典型用法（LLM视角）
用户：“打开百度，搜索Playwright MCP，截结果页”

LLM通过MCP自动执行：
1. `browser_navigate(url="https://www.baidu.com")`
2. 读快照 → 找到搜索框ref=e5
3. `browser_fill(ref=e5, text="Playwright MCP")`
4. 读快照 → 找到“百度一下”按钮ref=e8
5. `browser_click(ref=e8)`
6. 等待加载 → `browser_screenshot(path="result.png")`
7. 返回结果给用户
